# PushupAI 官网 Performance Editorial 视觉改版设计

**日期：** 2026-07-12
**状态：** 已确认
**设计方向：** A · Performance Editorial
**页面密度：** 精简重组为 6 个区块
**用户授权：** 后续细节采用推荐方案直接推进

## 1. 目标

在不改变官网既有产品事实、8 种语言、下载状态和隐私边界的前提下，重做官网视觉层级与响应式排版，使页面具备高端运动科技品牌的质感，并消除当前多语言文字纵向排列、移动端页面过长、留白失衡和正文过小等缺陷。

改版完成后，用户应当在首屏迅速理解三件事：

1. PushupAI 是使用手机摄像头进行端侧俯卧撑识别与计数的 App；
2. 原始视频帧不上传；
3. Google Play、App Store 与未来 APK 下载入口分别处于什么状态。

## 2. 当前页面已确认的问题

### 2.1 明确的纵向排版缺陷

当前 HTML 为可翻译文案增加了嵌套 `<span data-i18n>`，但 CSS 使用过宽的结构选择器：

- `.eyebrow > span`
- `.motion-label span`
- `.privacy-note span`

这些规则同时命中装饰圆点与翻译文本，把正常文案限制在约 `7–8px` 宽高内，导致中文、德语等语言逐字纵排，并显著拉长页面。改版必须给装饰节点单独 class，或仅选择 `[aria-hidden="true"]`，禁止依赖“所有 span 都是装饰”的假设。

### 2.2 视觉与信息结构问题

- 大面积浅绿背景贯穿全页，区块层级不够清楚；
- feature、showcase、ecosystem、steps、FAQ 之间重复解释，页面节奏松散；
- 卡片样式数量过多且阴影、圆角、深浅表面缺少统一层级；
- 桌面首屏缺少强品牌焦点，真实 App 截图没有形成主视觉中心；
- 移动端德语页面高度约 `10,157px`，阅读成本过高；
- 多处辅助正文低于移动端舒适阅读字号；
- 滚动显现动画会让未进入观察区的内容保持透明，不利于全页截图、快速滚动和稳定展示；
- 当前抽象背景素材存在，但与页面结构结合较弱，未形成独特品牌记忆。

## 3. 视觉方向

### 3.1 品牌气质

采用 Performance Editorial：克制、直接、有力量感，但不做电竞式霓虹堆叠。

- **主色：** Ink `#14231B`
- **强调色：** Acid Lime `#C6FF55`
- **功能绿：** Signal Green `#2AC76D`
- **主背景：** Warm Chalk `#F5F6F0`
- **卡片背景：** Paper `#FFFFFF`
- **辅助文字：** Slate Moss `#5E6D64`

颜色只承担层级和状态，不使用大面积高饱和绿。深色面用于核心能力、排行榜与下载区；酸性绿只用于焦点、状态和关键数字。

### 3.2 字体与排版

继续使用本地 system font stack，不引入外部字体请求。

- Hero 标题：`clamp(52px, 7vw, 104px)`，强字重，紧凑行高；
- Section 标题：桌面 `48–68px`，移动 `34–46px`；
- 正文：桌面 `17–19px`，移动端最低 `16px`；
- Label / eyebrow：`11–12px`，仅用于短标签，禁止承载长句；
- 英文、德语、葡萄牙语允许自然换行，不固定行数，不纵向排字；
- 正文行长桌面控制在约 `65ch`，移动端约 `35–55ch`。

### 3.3 表面、圆角与阴影

建立三级表面：

1. 页面底色；
2. 白色/深色内容卡片；
3. 手机设备与下载卡等高层级对象。

圆角统一为 `18 / 28 / 40px` 三档；阴影统一为轻、中、设备三档，不再为每个组件定义不同阴影。

## 4. 页面结构：6 个区块

### 4.1 Hero

- 左侧：品牌标签、三行主标题、真实产品说明、下载状态 CTA；
- 右侧：一台主手机 + 一台弱化的辅助手机，真实 App 截图；
- 背景：新生成的无文字 Performance Motion 视觉素材，与酸性绿轨迹叠加；
- 保留端侧隐私短说明；
- 商店按钮保留真实禁用状态，不伪装成可下载。

### 4.2 核心能力

三张能力卡：

1. 实时识别与计数；
2. 端侧隐私；
3. 周/月/年训练记录。

这一区块吸收旧 features 的全部核心内容，不重复解释。

### 4.3 App 界面

使用三张真实截图讲述“开始训练 → 实时识别 → 查看记录”的连续故事。桌面采用错落设备陈列，移动端采用单张主设备 + 横向可滑动次设备，避免三台手机同时挤压。

### 4.4 账号与生态

保留四项真实能力，但压缩为两行 editorial bento：

- 训练记录与云端同步；
- 日榜与周榜匿名排行；
- Google 账号与恢复购买；
- App 中文/英文界面和主题模式。

不加入价格、免费试用或尚未存在的功能承诺。

### 4.5 使用方式与 FAQ

桌面左右双栏：左侧三步开始，右侧 FAQ；移动端先三步，再 FAQ。保留相机摆放、端侧处理、动作范围、同步和下载状态等真实说明。

### 4.6 下载区

深墨色大卡片作为页面收束：

- Google Play：Alpha 封闭测试；
- App Store：准备中；
- APK：保持不可交互、不可扫描占位；
- 隐私政策和账号删除入口继续可访问。

## 5. ImageGen 素材设计

新增一个项目内素材 `website/assets/pushup-performance-motion-v2.webp`。

### 素材用途

Hero 与下载区的低对比运动背景，不承担产品信息，不包含文字。

### 画面要求

- 抽象、编辑感的俯卧撑运动轨迹；
- 人体只表现为几何/丝带式轮廓，不出现可识别人脸；
- Ink、Warm Chalk、Acid Lime 色系；
- 横向构图，右侧或中央偏右保留主视觉重心，左侧留出文案负空间；
- 无品牌 Logo、无文字、无水印、无真实健身房摄影感；
- 输出后转为 WebP，并控制文件体积。

## 6. 响应式规则

### 桌面（≥1024px）

- Hero 双栏；
- 最大内容宽度 `1240px`；
- 账号生态使用不对称 bento；
- 使用方式与 FAQ 双栏；
- 下载区文本与 APK 卡并列。

### 平板（768–1023px）

- Hero 保持双栏但缩小设备重叠；
- 主要网格改为两列；
- 下载区改为纵向主内容 + 横向渠道卡。

### 手机（≤767px）

- Hero 单栏，文案优先；
- 主标题不超过约 `15vw`；
- 正文最低 `16px`；
- 卡片全部单列或可控横向滑动；
- 语言选择器、导航、FAQ 点击区最小 `44px`；
- 不允许横向页面溢出；
- 页面目标高度显著低于当前德语版约 `10,157px`，避免重复区块造成的冗长。

### 横屏手机

- Hero 允许紧凑双栏；
- 避免使用固定 `100vh`；
- 导航和语言控件保持可操作；
- 下载卡不裁切二维码占位。

## 7. 动画与交互

- 删除“内容默认透明、等待 IntersectionObserver 才显示”的强依赖；
- 只保留 `180–300ms` 的 hover、press 与轻微进入动效；
- 动画仅使用 `transform` 和 `opacity`；
- `prefers-reduced-motion: reduce` 下禁用非必要动效；
- 手动语言切换继续更新 `?lang=` 与 `localStorage`；
- 多语言切换不得重建链接或改变下载安全边界。

## 8. 可访问性与内容边界

- 所有交互目标最小 `44×44px`；
- 正文与背景对比度满足 WCAG AA；
- 保留 skip link、可见 focus ring、语义 heading、原生 `<select>`、原生 `<details>`；
- 图片保留本地化 alt；
- APK 占位继续使用 `<article>`，不含 `<a>`、`button`、`href` 或假 URL；
- 无 JavaScript 时完整中文内容仍可阅读，语言选择器隐藏；
- 官网 8 种语言不暗示 App 也支持 8 种语言。

## 9. 验证标准

### 自动化

- 官网测试与 JavaScript 语法检查通过；
- 新增选择器边界测试，禁止装饰样式再次命中翻译文本；
- 新增六区块结构与新素材存在性测试；
- `git diff --check` 无问题。

### 浏览器 QA

对 `zh-CN / en / de / pt-BR / ja / ko` 至少抽查，并覆盖 `360 / 390 / 768 / 1024 / 1440px` 与手机横屏：

- 无横向溢出；
- 无逐字纵排；
- 无不可见 reveal 内容；
- 移动端正文 ≥16px；
- 语言切换、菜单、FAQ、深链接正常；
- APK 占位始终不可交互；
- 视觉截图确认 Hero、App 展示、账号生态、使用/FAQ、下载区层级清楚。

## 10. 非目标

- 不修改 Flutter App；
- 不修改 Worker、账号、会员或商店配置；
- 不增加真实 APK、二维码或商店 URL；
- 不引入外部字体、前端框架或分析脚本；
- 不重写现有多语言数据模型。
