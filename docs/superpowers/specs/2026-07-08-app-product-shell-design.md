# App 产品化界面与训练闭环设计

日期：2026-07-08  
范围：把现有俯卧撑计数 Demo 包装成第一版成熟产品体验，同时保留 Demo 测试入口。

## 1. 背景

当前 App 已调通端侧 MoveNet 推理和俯卧撑计数核心能力。现有 `HomePage` 是工程 Demo：离线回放和实时相机两个 Tab。下一阶段目标是让普通用户打开 App 后直接进入清晰的训练路径，而不是看到开发测试面板。

本设计遵循第一性原则：第一版产品只闭合“开始训练 -> 识别准备 -> 计数播报 -> 结束保存 -> 查看记录”这条主链路。登录、同步、成就、组数系统等功能不进入本轮。

## 2. 已确认决策

- 方案选择：产品壳 + 训练闭环。
- 首页右上角计数板显示“今日总数”，点击进入日历。
- 左上角个人入口做可点击占位页，不做登录和资料编辑。
- 底部测试按钮进入当前 Demo 双 Tab 页面。
- 正式训练页前期保留骨架/关键点叠加。
- 准备状态第一版只判断人进入画面、头肩关键点可见且稳定、位置合理；不做严格动作标准判断。
- 训练结束后无论计数多少都保存 session，包括 0 个。
- 计数播报只播报 1 到 30；超过 30 继续计数但不播报。
- 音频用 MiMo 预生成固定资源，App 播放本地音频。
- MiMo 音色使用 `冰糖`，风格为清晰、轻快、鼓励但不过度兴奋。
- 视觉方向采用“冷静训练教练”：圆润、明亮、友好，但不过度游戏化。

## 3. 页面结构

### 3.1 正式首页

`HomePage` 改为正式产品首页，不再直接显示 Demo Tab。

布局：
- 顶部左侧：个人信息入口图标，点击进入占位页。
- 顶部右侧：今日计数板，显示当天所有训练 session 的计数汇总，点击进入日历记录页。
- 中央：圆形开始按钮，点击进入正式训练页。
- 底部：测试模式按钮，点击进入现有双 Tab Demo。

### 3.2 正式训练页

`WorkoutPage` 负责普通用户训练：
- 展示相机画面。
- 保留骨架/关键点叠加。
- 展示当前训练状态和当前计数。
- 提供“结束训练”按钮。

### 3.3 记录页

`RecordsPage` 展示日历式训练记录：
- 以本地日期为单位汇总 session。
- 每天显示当天总俯卧撑数量。
- 第一版不展示 session 明细列表；只做日历日汇总。

### 3.4 个人占位页

`ProfilePlaceholderPage` 只做未来同步能力入口预留：
- 头像/昵称占位。
- 同步能力提示。
- 不做登录、注册、头像上传、资料编辑。

### 3.5 测试模式页

`TestModePage` 复用当前 Demo：
- `OfflineReplayTab`
- `LiveCameraTab`

测试页风格不要求和正式产品统一，保持能开发测试即可。

## 4. 训练状态机

训练流程：

1. `loading`：加载模型和相机。
2. `guide`：播放摆放引导语。
3. `waitingReady`：读取实时关键点，等待准备状态。
4. `counting`：准备完成后重置计数器并开始实时计数。
5. `saving`：用户点击结束训练，停止相机并保存 session。
6. `saved`：返回首页并刷新今日总数。

音频规则：
- 进入训练页并启动相机流程后，播放：
  - “请保持俯卧撑姿势，并且把手机放在正前方，正对头顶肩部”
- 准备状态通过后，播放：
  - “您已进入准备状态，请开始您的俯卧撑训练，我会为您计数”
- 每次计数从 `N-1` 变为 `N`，且 `1 <= N <= 30`，播放数字 `N`。
- `N > 30` 后继续计数但静音。
- 音频播放需要避免重叠；同类提示不重复排队。

准备状态第一版规则：
- 头部和双肩关键点置信度达到 `0.3` 初始阈值。
- 头肩中心位于画面中部合理范围内，初始采用画面宽高的中间 `80%` 区域。
- 关键点连续约 `0.5s` 稳定，按时间判断，不绑定固定帧数。
- 通过后才 reset 并启用 `PushupCounter`，避免摆手机和入镜动作被计入训练。

## 5. 数据设计

本轮只保存本地训练 session，不保存视频、帧图、关键点历史或生物特征。

数据模型：

```dart
class WorkoutSession {
  final String id;
  final DateTime startedAt;
  final DateTime endedAt;
  final int count;
}
```

存储方案：
- 使用 App documents directory 下的 JSON 文件。
- 文件名为 `workout_sessions.json`。
- 新增 session 时读出列表、追加、写回。
- 第一版数据量很小，不引入数据库。

汇总规则：
- 首页今日总数：按本地日期过滤当天 session，求 `count` 总和。
- 日历页：按本地日期分组，求每天 `count` 总和。

## 6. 音频资源

资源来源：
- 使用 `C:\Users\29485\.pi\agent\tasks\subtitle-to-speech` 通过 MiMo TTS 生成。
- 运行时依赖 `MIMO_API_KEY`，不得在代码、日志、文档或回复中暴露完整 key。

需要生成的固定资源：
- 摆放引导语。
- 准备完成语。
- 数字 `1` 到 `30`。

Flutter 播放：
- 优先使用项目已有能力；如果没有，新增一个最小本地音频播放依赖。
- 封装为 `VoicePromptPlayer`，正式训练页只调用语义方法，不直接管理文件名和播放队列。

## 7. 视觉方向

参考多邻国的产品感，但不复制品牌、吉祥物或具体图形。

方向：
- 明亮但不花哨。
- 圆润控件。
- 训练页以相机区域为中心。
- 使用蓝/绿/黄等积极色，但整体更像“训练教练”而不是儿童游戏。
- Demo 页不强制重做视觉。

## 8. 验收标准

必须通过：
- App 首屏为正式首页，不是 Demo Tab。
- 首页开始按钮进入正式训练页。
- 训练页能启动相机和模型。
- 摆放引导音频播放一次。
- 准备状态通过后播放准备完成音频。
- 实时相机流接入 `PushupCounter`，能从真实训练中计数。
- 计数 `1` 到 `30` 播放本地数字音频。
- 超过 30 继续计数但不播报。
- 点击结束训练后保存 session 并返回首页。
- 首页今日总数和日历日汇总都来自保存的 session。
- 测试模式仍能打开当前离线回放和实时相机 Demo。
- `flutter analyze` 通过。
- `flutter test` 通过。

建议补充的最小测试：
- session JSON 存储读写。
- 按本地日期汇总计数。
- 准备状态判定的稳定/不稳定样例。

## 9. 非目标

本轮不做：
- 登录、注册、云同步。
- 个人资料编辑。
- 成就、连续打卡、等级、组数系统。
- 严格动作标准判断。
- 视频、帧图、关键点历史保存。
- 吉祥物或复杂美术资产。
- Demo 页完整重设计。

## 10. 对抗式审查结论

主要风险与处理：

- 风险：当前实时相机 Demo 只画骨架和性能数据，没有把 `PushupCounter` 接入正式训练流。  
  处理：正式训练页必须把实时关键点转信号并调用同一个 `PushupCounter`，不要复制算法。

- 风险：准备阶段动作被误计数。  
  处理：Ready Gate 通过后再 reset 并启用计数器。

- 风险：音频播放重叠导致体验混乱。  
  处理：统一封装 `VoicePromptPlayer`，限制提示重叠。

- 风险：为了“成熟产品”过早加入账户、成就、同步。  
  处理：本轮只闭合训练和记录主链路。

- 风险：产品路径保存隐私敏感数据。  
  处理：正式路径只保存 session 元数据和计数，不保存视频、图像和关键点历史。
