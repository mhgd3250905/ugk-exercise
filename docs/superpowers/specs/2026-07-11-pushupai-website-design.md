# PushupAI 官网设计规格

日期：2026-07-11
状态：已确认视觉方向，待实现
范围：`website/` 独立静态产品官网

## 1. 目标

为俯卧撑计数 App 建立一个可直接部署的官方产品网站。网站以 App 本身和未来的 Google Play、App Store 下载渠道为核心，传达成熟、简约、有质感的产品形象，并与现有 Flutter App 的视觉和真实能力保持一致。

品牌名：

- 中文：`AI俯卧撑`
- 英文：`PushupAI`

网站第一版不提供账号登录、购买、后台管理或动态内容系统。

## 2. 受众与成功标准

主要受众是希望用手机辅助完成标准俯卧撑训练的普通用户。

成功标准：

- 首屏能在几秒内说明产品用途：架好手机、进入姿态、自动识别并计数。
- 真实展示 App，而不是使用虚构界面或未落地功能。
- 清楚呈现 Google Play 和 App Store 两个下载渠道；未上架前统一显示“即将上架”。
- 手机、平板和桌面浏览器均可正常阅读和操作。
- 下载链接上架后可以集中替换，不需要改动页面结构。
- 可以作为独立静态站部署到 Cloudflare Pages、GitHub Pages 或其他静态托管服务。

## 3. 视觉方向

采用已确认的 A 方向：产品编辑感。

- 延续 App 的浅绿色、米白、深墨绿和亮绿色品牌色。
- 使用大字号标题、宽松留白、圆角卡片、精细边框和克制阴影。
- Hero 以真实 App 手机画面为视觉中心，不使用通用健身房照片。
- 可使用一张由 `imagegen` 生成的无文字抽象运动背景作为氛围素材，但不得让它抢过 App 主体。
- 动效只服务于层级和浏览节奏，不使用连续弹跳、炫光或强烈视差。
- 避免紫色科技渐变、黑金健身房风和与 App 不一致的霓虹视觉。

## 4. 内容与信息架构

### 4.1 顶部导航

- 品牌组合：`PushupAI` + `AI俯卧撑`
- 锚点导航：产品能力、使用方式、下载
- 主行动按钮：`即将上线`
- 移动端折叠为简洁菜单，保持键盘可操作。

### 4.2 Hero

主标题：`架好手机，专心做好每一次。`

说明：`端侧 AI 实时识别动作，自动计数、语音播报，并为你留下每一次训练。`

操作区：

- Google Play 下载占位按钮
- App Store 下载占位按钮
- 两者当前显示“即将上架”，不跳转到空链接

视觉区使用现有 App 首页与训练页截图，置于代码实现的手机外框中。抽象运动背景仅作为低对比度衬托。

### 4.3 核心能力

只展示当前已落地能力：

1. `看见动作，自动计数`：MoveNet 姿态识别与俯卧撑计数。
2. `训练留在你的设备`：端侧推理，视频帧不上传。
3. `每次进步，都有记录`：中文语音播报与本地训练记录。

不宣传未经证实的准确率、跨运动识别、社交挑战或健康诊断能力。

### 4.4 产品展示

使用现有真实截图展示：

- 首页：进入训练和今日完成量。
- 训练页：相机姿态识别、准备状态与大号计数。
- 记录页：训练日历和训练汇总。

桌面端使用错落手机陈列；移动端改为横向滚动或单列，保证截图可读。

### 4.5 使用流程

三个步骤：

1. 固定手机并让身体进入画面。
2. 保持标准俯卧撑姿态，等待识别准备完成。
3. 开始训练，由 App 自动计数并语音播报。

附带轻量提示：单人、手机固定正前方、标准宽距俯卧撑、光线充足。

### 4.6 下载区

使用高对比度深墨绿区域收束页面，重复品牌主张和两个商店入口。

下载 URL 集中定义在 JavaScript 配置对象中。值为空时按钮显示“即将上架”并保持不可跳转；填入 URL 后自动变为真实链接。

### 4.7 页脚与隐私

- 品牌名和一句简短产品说明。
- 隐私要点：姿态识别在设备端完成，视频帧不上传。
- 版权年份自动生成。
- 暂不创建虚假的社交媒体链接或未完成的法律页面。

## 5. 技术设计

在仓库根目录新增 `website/`：

```text
website/
├── index.html
├── styles.css
├── main.js
├── README.md
└── assets/
    ├── app-home.png
    ├── app-workout.png
    ├── app-records.png
    └── pushup-motion-bg.webp
```

技术约束：

- 原生 HTML、CSS、JavaScript，不引入框架、包管理器或运行时依赖。
- 使用语义化标签、可见焦点、正确的按钮/链接语义和合理的颜色对比度。
- 图片提供尺寸属性、响应式尺寸和延迟加载；首屏关键图除外。
- JavaScript 只负责移动菜单、下载链接状态、渐进式入场和年份，不承载页面主体内容。
- 在 JavaScript 不可用时，核心文案、截图和下载状态仍然可见。
- 支持 `prefers-reduced-motion`；不强制实现另一套深色视觉，优先保证与 App 浅色品牌一致。

## 6. 素材策略

- App 截图复制自 `docs/design/assets/`，保留真实界面，不直接引用文档相对路径。
- `imagegen` 生成一张项目专用背景素材，分类为 `stylized-concept`。
- 生成内容为无文字、无 Logo、无设备 UI 的抽象俯卧撑运动轨迹；配色限定为浅绿色、青绿色、柠檬绿和米白。
- 输出进入 `website/assets/`，网站不引用 `$CODEX_HOME` 下的临时文件。
- 若生成素材不适配页面，允许不使用，但需保留真实 App 截图作为完整视觉方案。

## 7. 响应式与交互

建议断点：

- `>= 1024px`：双栏 Hero、三列能力卡、三台手机陈列。
- `640px–1023px`：收紧双栏或改为上下结构，两列内容卡。
- `< 640px`：单列结构、折叠导航、下载按钮全宽、截图保持可滑动或单列。

交互原则：

- 页面锚点平滑滚动，但在减少动画模式下关闭。
- 入场动画使用小幅透明度和位移变化，只播放一次。
- 所有交互控件具备 hover、focus-visible 和 active 状态。
- “即将上架”渠道不能伪装成可用下载链接。

## 8. 错误与降级

- JavaScript 未加载：页面内容完整可读，下载按钮保持静态“即将上架”。
- 生成背景加载失败：布局和文字不受影响，由 CSS 背景色承接。
- App 截图加载失败：保留有意义的 `alt` 文本和稳定的手机框架尺寸。
- 下载 URL 配置错误或为空：不导航，继续显示未上架状态。

## 9. 验证

实现后至少完成：

- HTML 结构与本地资源引用检查。
- JavaScript 语法检查。
- 本地 HTTP 服务下的桌面与移动端浏览器截图检查。
- 360px、768px、1440px 三种宽度的横向溢出检查。
- 键盘导航、焦点状态和减少动画模式检查。
- 下载渠道为空时不可跳转；配置测试 URL 后可以正确跳转。
- Lighthouse 或等价检查重点关注 accessibility、SEO 和静态资源加载。
- 确认没有修改 Flutter 业务层，也没有引入凭证、真实视频或隐私数据。

## 10. 明确不做

- 不接入真实商店链接，因为当前尚未提供。
- 不制作会员购买 Web 流程。
- 不建立 CMS、博客、账号中心或数据收集表单。
- 不虚构 iOS 已可用、识别准确率或未来功能。
- 不调整 Flutter App 的算法、页面或会员系统。
