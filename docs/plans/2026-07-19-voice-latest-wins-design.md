# 语音最新事件抢占与英文数字倍速设计

## 用户问题

快速俯卧撑时画面计数正确，但数字音频按队列等待上一段自然结束，导致动作完成 15 次后仍继续播报 13、14、15。英文数字素材时长还会进一步放大该滞后。

## 决策

1. 所有 guide、ready、count 语音统一采用 latest-wins：新事件到来后，旧事件立即失效，当前音频停止并切换到最新事件。
2. 只串行执行播放器的 stop/play/rate 控制，绝不等待 WAV 播放完成；用 generation 守卫阻止旧异步调用覆盖新事件。
3. 英文数字计数使用 1.2 倍速；英文 guide/ready 与全部中文语音使用 1.0 倍速。每条音频播放后显式设置速度，防止播放器状态残留。
4. 保留原有 1—30 播报范围、音频素材、启动预加载与 Controller 调用方式；不改识别、计数和训练状态机。

## 放弃的方案

- 保留数字队列，仅对英文加速：1.2 倍仍可能慢于极快动作，无法消除累计滞后。
- 只让数字互相抢占：会留下 guide/ready 与 count 之间不一致的优先级语义。
- 离线重采样英文 WAV：会修改 30 个素材且难以快速试验倍率；运行时倍速更小、更易回滚。

## 验证

- 先用阻塞播放器复现第二个数字无法及时启动，确认旧实现测试失败。
- 验证英文 count 为 1.2 倍、英文 ready 为 1.0 倍、中文 count 为 1.0 倍。
- 验证前一数字保持 playing 时，后一数字无需等待即可启动。
- 运行语音定向测试、`flutter analyze`、全量 `flutter test` 和 `git diff --check`，再做真机快速动作听感验收。
