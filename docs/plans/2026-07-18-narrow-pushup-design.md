# 窄距俯卧撑功能设计

日期：2026-07-18

## 目标与成功标准

在不改变常规俯卧撑默认路径和 step0/v3/v4 = 5/5/3 回放基线的前提下，新增可从首页独立进入的“窄距俯卧撑”训练类型。窄距模式复用现有相机、准备态、头肩位移计数、语音、训练页和记录链路，只增加准备态与每次回顶时的窄距形态资格门控。

成功标准：

1. 首页中英文均显示常规与窄距两张训练卡，分别进入显式运动类型驱动的同一训练页。
2. 常规俯卧撑不调用窄距形态门控，现有行为、测试和 5/5/3 回放不变。
3. 窄距准备态只有在严格支撑姿态和窄距形态均可靠可见、均通过时才能进入 ready。
4. 窄距每次动作只有在完成现有下压—回顶闭环且顶部窄距形态通过时计数；底部肘腕短暂遮挡不丢失动作。
5. 明显宽距、低置信、退化几何和整体相机平移不能绕过窄距门控；快速窄距动作仍能计数。
6. 本地记录、云同步请求与云历史保留 `exerciseType`；记录页聚合同单位 reps，首页卡片按类型统计。
7. 诊断日志记录运动类型、窄距判据结论和尺度无关标量，不增加原图、视频、音频或设备标识。

## 算法可行性

MoveNet COCO-17 在正面顶部支撑姿态可提供肩、肘、腕二维关键点，因此能观察“手臂是否明显向外张开”的屏幕投影。它不能观察手掌接触、真实三维深度、手型或精确握距，所以本功能只能实现保守资格门控，不能宣称可靠分类所有普通/窄距姿势。

单一“前臂方向差”不够：肩宽支撑时两条前臂也可能近似平行。单一固定像素腕距也不够：会随人物远近改变。采用三个互补、尺度无关的顶部标量：

- `wristSpanRatio = |leftWrist.x - rightWrist.x| / shoulderSpan`
- `elbowSpanRatio = |leftElbow.x - rightElbow.x| / shoulderSpan`
- `forearmDirectionDeltaDegrees`：左右“肘→腕”向量相对画面竖直向下方向的角度差

首轮容差采用 `wristSpanRatio <= 1.15`、`elbowSpanRatio <= 1.35`、方向差 `<= 30°`。六个肩/肘/腕关键点均须达到 0.3；肩宽、前臂方向不可计算或腕未可靠位于肘下方时结论为 `unknown`，绝不当作通过。阈值用于拒绝明显外张，并非人体运动学真理，最终需要窄距/普通对照真机日志校准。

三个标量对整体平移、镜像和等比缩放不变；轻微左右不对称由比值和 30° 容差吸收。肩宽但双臂竖直的普通支撑仍可能通过，这是单目 2D 和保守容差的已知假阳性边界。

## 状态与数据流

新增产品枚举 `ExerciseType.pushup` / `ExerciseType.narrowPushup`，持久化值分别为 `pushup` / `narrow_pushup`。

准备阶段：

1. 现有 `ReadyPoseGate` 严格检查核心点、双腕双髋、腕髋关系、居中和稳定 500ms。
2. 常规类型直接执行现有深度标定。
3. 窄距类型额外评估顶部形态。`matches` 才标定；`doesNotMatch` 或 `unknown` 重置准备稳定窗口，并显示窄距调整提示。

运动阶段：

1. 现有 `motionPoseUsable` 与 `PushupPipeline` 继续只用可信头肩轨迹推进动作，底部肘腕可以离屏。
2. 窄距类型每帧计算形态结论，并把它映射为通用回顶决策：通过、拒绝、等待。
3. `PushupCounter` 只在现有回顶完成条件成立时读取该决策：通过则沿用肘角反证后计数；拒绝则结束本次但不计数；等待则保持当前 dip，等待后续顶部可靠帧。
4. 常规类型始终传入默认“通过”，保持原 API 行为和相位机路径。

## 产品与合同边界

- 首页两张卡分别展示当日该类型次数；顶部今日汇总和记录日历合并两种 reps。
- `WorkoutPage` 保存显式类型，`WorkoutController` 仅负责识别编排，不写存储。
- 云同步与 D1 现有 schema 已是通用 `exercise_type TEXT`，无需 migration；本地 Worker 合同增加 `narrow_pushup` 白名单。排行榜 Controller 仍固定查询 `pushup`，窄距成绩不会混入常规榜单。
- Worker/D1 只做本地代码和测试，不部署、不迁移、不写远程数据。
- 语音继续复用 1—30 数字素材，不新增运动类型语音。

## 验证策略

- 纯算法：窄距、明显宽距、轻微不对称、低置信、退化几何、平移/缩放/镜像不变性。
- 计数：宽距顶部拒绝、低置信顶部等待后恢复、底部手臂离屏、快速三帧回顶、常规默认路径不变。
- Controller：窄距准备失败不 ready，窄距通过可 ready；日志含类型和标量。
- UI/存储：两张卡、路由类型、中英文、窄距 session 保存、按类型当日统计、记录聚合。
- 云合同：Flutter 请求/响应接受 `narrow_pushup`，Worker 同步与真实 SQLite 按独立 `exercise_type` 保存/聚合，未知类型仍拒绝。
- 全量：`flutter analyze`、`flutter test`、Worker `npm test`、`git diff --check`，并从测试输出核对 step0/v3/v4 = 5/5/3。
- 真机首轮对照（2026-07-18）：正面固定机位下，窄距模式严格准备后连续 5 次均计数；随后故意放宽双手的动作不计，恢复合格姿势后继续计为第 6 次。结构化轨迹显示第 5—6 次间隔 18 秒，中间 388 帧无计数增量，其中 238 帧明确为 `doesNotMatch`；第 5、6 次计数帧均为 `matches`，全程无 frame error。常规模式以稍宽手位完成 3 次，日志类型为 `pushup` 且连续计为 1—3。
- 真机仍待补充：近远景、轻微透视、快速动作、底部遮挡、多用户/机型样本和 Release 诊断导出。首轮单用户单机位通过不等于真实场景准确率验收完成。
