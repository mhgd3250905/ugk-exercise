## 1. Review 总目标与成功标准

**总目标**：产出一本"接手者生存手册"级别的 review 报告，回答三个问题：
1. 一个全新的人/agent clone 代码后，**多久能独立跑起来并改第一个功能**？卡点在哪？
2. 项目的**哪些地方一旦改动会牵连很多、风险高**？哪些是安全区？
3. 文档与代码**哪些地方已经对不上**，会误导接手者？

**成功标准**（review 完成时应交付）：
- 一份按维度组织的发现清单，每条标注：维度 / 严重度 / 证据（文件:行）/ 复核状态（仍存在/已修复/部分）/ 建议（不改代码，只记录）。
- 一份"接手者最先该读什么/跑什么"的索引建议。
- 明确标注哪些是**故意的设计保留项**（不当作问题），哪些是**真欠账**。

**范围声明**：本 review 是**只读分析**，不改产品代码。发现的问题在报告里记录，由后续单独授权处理。

## 2. 十三个 Review 维度（按可接手性优先级排序）

> 优先级 P0 = 直接决定"能不能接手"；P1 = 决定"接手后会不会踩坑"；P2 = 长期可维护性。

### 维度 1（P0）：环境可复现性与"跑起来"门槛
**为什么**：四篇审计里"干净 checkout 无法复现测试"（`俯卧撑.mp4`、`step0/*.csv` 未入 git）是被红队当 bug 发现的，但**从没有正面的环境准备清单**。这是可接手性的第一道门。

**Review 内容**：
- 从零 clone 到 `flutter test` 全绿，需要哪些**不在 git 里**的前置（tflite model、回放夹具、`android/key.properties`、dart-define 配置、本地音频资产）？逐一列出处和获取方法。
- `test/fixtures/` 是否自洽？干净 checkout 能否跑通回放基线 step0=5/v3=5/v4=3？
- 是否存在 README 或文档说明"如何本地运行"？`README.md` 当前内容是否覆盖这些？
- 模拟器/真机运行的最低要求（API level、ABI、是否有内置 tflite delegate 限制）。

**方法**：理论推演 + 实际在干净状态下尝试（若条件允许）。

### 维度 2（P0）：文档与代码的真实性对齐（失真审计）
**为什么**：audit-2026-07-13 维度 4 已发现测试数/提交数/历史基线绑定等失真，但当时基线已滞后。0.3.8 之后又有 15+ 提交，失真可能扩大。**接手者读到的文档如果是错的，会建立错误的心智模型。**

**Review 内容**：
- AGENTS.md 的测试数（写 363）、版本基线描述与实际（flutter test 407、worker 138）是否对齐。
- `docs/release-configuration.md` 前后矛盾（audit 指出"前部已部署、后部仍列为待办"）在 0.3.8 后是否仍矛盾。
- `docs/architecture-analysis.md` / `architecture-plan.md` 是否加了"历史基线"横幅（audit 建议过）。
- `docs/modules/membership.md` 是否还是 2026-07-09 旧分支汇报（audit 指出过）。
- 各文档引用的文件路径/行号/提交 hash 是否仍有效（handoff 引用失效是 audit 发现的）。

**方法**：逐份文档核对关键事实，列出失真项 + 应更新成的值。

### 维度 3（P0）：计划文档的状态治理（29 份 plans 的可接手性）
**为什么**：`docs/plans/` 有 **29 份计划文档**，audit 指出"多份已落地计划无完成状态"。接手者无法判断"这个 plan 是已实现、废弃、还是待办"——这是**纯可接手性灾难**。

**Review 内容**：
- 29 份 plan 逐一标注：已落地（对应哪个提交）/ 部分落地 / 废弃 / 待定。
- 是否需要一个 plan 索引（README 或 index 文件）。
- `docs/reviews/`、`docs/refactor/`、`docs/reports/`、`docs/superpowers/plans/` 重复的规划类文档如何归并/标注。

**方法**：逐份读 plan + 用 git log 对照是否对应实际提交。

### 维度 4（P1）：Worker 子系统的可接手性
**为什么**：Worker 是独立 TS 子系统（14 文件/2724 行），但前三篇审计**完全没提它**，audit 才首次顺带验证安全。Flutter 与 Worker 的契约、D1 schema、部署流程都未文档化为可接手维度。

**Review 内容**：
- Worker 目录结构与职责划分是否清晰。
- **Flutter↔Worker API 契约**：端点列表、请求/响应 schema、错误码映射，是否有单一事实源（还是散落在 client 代码 + 路由代码两处）。
- D1 schema 与 migration 流程（0001–0005）的可接手性：新加字段的流程、回滚约束。
- 跨 Dart/TS 的类型一致性（同一概念两边各定义一次？）。
- 部署流程文档化程度（本会话发现 Token 权限不足、Secret 来源缺失，正说明部署知识传承有缺口）。

### 维度 5（P1）：测试策略矩阵
**为什么**：四篇都引用测试数和回放基线，但**从没有"测什么/不测什么"的策略**。接手者不知道改了 X 该跑什么、哪些路径明确不测、回放基线守的是什么。

**Review 内容**：
- 测试分层现状：unit（domain/product）/ widget / contract（architecture_contract_test）/ 回放基线，各自守什么。
- **覆盖度盲区**：audit M1 指出 controller 竞态缺测试，refactor-report 指出"异步生命周期/真机边界未覆盖"——这些是否仍缺。
- Fake/依赖注入基建：Controller 无法注入 fake 被列为设计问题（refactor-report 6.6），现状如何。
- 回放基线 step0=5/v3=5/v4=3 守护的是什么、改动信号源的边界。
- 改各类代码的"该跑什么"速查（接手者最需要的）。

### 维度 6（P1）：数据模型与持久化契约
**为什么**：本项目用 `path_provider` 文件 + `flutter_secure_storage`（无 sqflite/hive），本地↔D1 云端同步存在（workout_sync_controller），但**没有任何一篇作为维度覆盖**数据契约。

**Review 内容**：
- `WorkoutSession` 等核心数据结构定义与演进。
- 本地存储格式（文件布局、序列化方式）。
- 本地↔云端同步的冲突解决、离线优先策略、owner 绑定。
- 数据迁移路径（schema 变了老数据怎么办）。

### 维度 7（P1）：错误处理与降级总则
**为什么**：audit L2 只列了 3 处空 catch，但**没有全仓的"错误如何分层"总则**。`workout_page` 重试保存是个例不是体系。接手者改代码时不知道"这里该吞还是该抛"。

**Review 内容**：
- 全仓错误处理模式梳理：哪些吞（空 catch 降级）、哪些抛、哪些给用户可见反馈。
- 是否有一致的错误呈现策略（toast/snackbar/状态码）。
- 重试策略（哪些可重试、退避）。
- audit L2 的"降级路径缺诊断可观测性"是否仍是欠账。

### 维度 8（P1）：依赖与版本治理
**为什么**：完全未涉及。Flutter SDK `^3.8.0`、`tflite_flutter ^0.12.0`、`camera ^0.11.0`、`purchases_flutter ^10.4.1` 等，**升级策略和锁定策略**都没文档。

**Review 内容**：
- 关键依赖（tflite/camera/RevenueCat/google_sign_in/audioplayers）版本与已知风险。
- 是否锁定版本（`^` caret 的范围）、升级策略。
- Worker 只有一个运行依赖 `jose`，是否最小化。
- 依赖漏洞面（重点 tflite/RevenueCat 这类涉及支付/推理的）。

### 维度 9（P2）：l10n / 国际化可维护性
**为什么**：audit M2 发现"中文字符串当状态码"问题。i18n 工作流（ARB 维护、加语言、状态映射完整性如何守）未系统说明。

**Review 内容**：
- ARB 维护流程，新增文案的步骤。
- audit M2（字符串状态码）现状复核。
- `app_en.arb` / `app_zh.arb` 的键是否对齐、有无遗漏。
- 状态码映射的完整性守护。

### 维度 10（P2）：CI/CD 与发布流水线
**为什么**：**无 CI**（无 `.github/workflows`），验证全靠手工命令。这是可维护性欠账——没有自动门禁，回归全靠人记得跑。

**Review 内容**：
- 评估引入 CI 的成本/收益（至少 analyze + test + 回放基线自动跑）。
- 版本号管理（`0.3.8+11` 的来源、versionCode 递增规则是否有文档）。
- release 文档（release-configuration.md）与实际流程的对齐。

### 维度 11（P2）：权限与隐私合规
**为什么**：相机、网络（Worker/RevenueCat/Play）、本地存储权限，四篇审计**均未提**。audit 安全维度只查凭证不查权限。

**Review 内容**：
- AndroidManifest 权限声明清单与必要性。
- 数据收集范围与隐私政策对齐（官网 pushupai.pages.dev 的隐私声明 vs 实际收集）。
- 相机数据处理（是否留存、是否上传）。

### 维度 12（P2）：可观测性与线上排障
**为什么**：UGK 诊断日志存在（`adb logcat -s flutter | grep UGK`），但**没有日志策略文档**。线上出问题怎么排障、日志格式/级别/采样、crash 收集都未覆盖。

**Review 内容**：
- UGK tag 覆盖的事件、日志格式约定。
- 线上 crash/异常收集（是否有接入，还是纯本地 logcat）。
- 排障 SOP（文档化程度）。

### 维度 13（P2）：性能预算与基准
**为什么**：perf 模块存在、refactor-report 提到 `pass` 字段混淆，但**没有性能基线**（推理延迟/FPS/内存目标值）。

**Review 内容**：
- 推理延迟/FPS 的现状实测与目标。
- perf 模块的用途与 `pass` 字段语义。
- 是否需要建立性能回归门禁。

## 3. 执行方法与产出

### 执行原则
- **只读**：不改产品代码、不部署、不上传、不动 36 个未跟踪文件。
- **对抗式**：不轻信文档自述，去代码核对（沿用本项目审核传统）。
- **可追溯**：每条发现标注证据（file:line）和复核状态。

### 产出物
1. **主报告**：`docs/reviews/2026-07-16-full-review-report.md`——按 13 维度组织，每条发现含维度/严重度/证据/复核状态/建议。
2. **接手者索引**（报告附录或独立文件）：新接手者最先该读的 5 份文档 + 最先该跑的命令 + 最该小心的 3 个高风险区。
3. **失真文档更新清单**：列出该改的文档 + 应更新成的值（更新动作本身需单独授权）。

### 严重度定义
- 🔴 **阻塞接手**：不解决，新接手者无法跑起来或会被严重误导。
- 🟡 **增加接手成本**：能接手但会踩坑、浪费时间。
- 🔵 **长期可维护性**：短期不痛，长期债务。

### 既有审计条目的复核优先级
执行时先快速复核 audit-2026-07-13 的 M1–M3/L1–L3 在当前代码（`d9533fd`）是否仍存在，纳入结论，避免遗漏已发现的问题：
- M1 `switchCamera()` session 守卫时序（`workout_controller.dart`）
- M2 中文字符串状态码
- M3 `replay_utils.dart` 反向 import `app_theme.dart`
- L1 匿名头像 UI 死回退
- L2 降级路径缺可观测性
- L3 长 build 方法
