我对原 `docs/audit-2026-07-13.md` 中的结论逐项回到代码、测试、项目规范和本机公开/脱敏发布台账核验。复核发现：原报告保留了正确的自动化基线，但部分问题把“工程偏好”写成了“架构违规”，部分风险场景描述与真实控制流不符，并且对安全和发布状态作了超出证据范围的绝对判断。

## 二、总体判断

`1531c37` 的本地自动化基线健康：Flutter 静态分析无 issue，Flutter 测试 312/312、Worker 测试 106/106，算法回放保持 step0=5 / v3=5 / v4=3。

本次确认的主要工程债务是：

1. `switchCamera()` 的 session 守卫晚于多个 `await`，需要先补 stop/dispose 竞态测试再修。
2. WorkoutController 用中文字符串充当内部状态码，当前 UI 能本地化，但合同脆弱。
3. 资源常量位置同时受到相互矛盾的分层原则和开发指南约束，应先统一规则。
4. UI 中存在基本不可达的匿名头像回退算法，可以删除。
5. README、AGENTS、membership、release 等文档存在明显过期或前后矛盾。

本次没有证据支持“存在已复现功能回归”；同样也不能从本地测试全绿推导“没有未知 bug”“安全事项全部闭环”或“发布链路没有阻塞”。正式购买链路和疑似暴露 Token 的轮换事项仍应以发布台账的 `BLOCKED` / P0 状态为准。

## 四、修订中补充发现的问题

原报告还存在以下内部或证据边界问题，我已一并纠正：

- 一处称匿名头像回退为死代码，另一处又概括“无死代码”，前后矛盾。
- 一处把 product/domain 描述为纯净，但 product 当前实际依赖 `audioplayers`、`dart:io` 和 `path_provider`；同时 AGENTS 又把语音和存储列在 product，属于架构定义自身不一致。
- “无安全泄露、无发布阻断”超出了本地静态检查的证明范围。仓库扫描没有发现真实凭证值，只能支持这一有限结论。
- README/AGENTS 中测试数量已滞后；`v0.4-reproducible` 到当前基准实际为 132 个提交，不是“约 30 个”。
- membership 文档仍是旧阶段汇报；release 文档前部记录相关能力已部署，后部仍把其中一项列为待办。
- 历史架构文档绑定旧提交基线，不应直接改写成当前事实；应标记为历史基线或另写当前概览。
