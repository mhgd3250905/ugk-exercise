## 1. 范围与不做项

### 1.1 本版交付
- Android 单平台的 Flutter App，实时摄像头俯卧撑计数
- MoveNet Lightning 本地推理，纯端侧，视频帧不上传
- 标准俯卧撑（手臂与身体两侧垂直，肘外展）的**完整上下循环计数**

## 2. 系统架构

### 2.1 分层模型
```
┌─────────────────────────────────────────────┐
│  UI Layer (overlay_view, session widgets)    │
├─────────────────────────────────────────────┤
│  Domain Layer (PushupCounter, SignalFilter)  │  ← 纯 Dart,无 Flutter/相机依赖,可单测
├─────────────────────────────────────────────┤
│  Inference Layer (PoseEstimator, TFLite)     │
├─────────────────────────────────────────────┤
│  Pipeline Layer (FramePipeline: YUV→RGB...)  │
├─────────────────────────────────────────────┤
│  Platform Layer (CameraService, delegate)    │
└─────────────────────────────────────────────┘
```

**核心约束**：Domain Layer（状态机/滤波/信号提取的纯算法）**必须零平台依赖**，只吃 `List<KeyPoint>` 吐 `CounterState`。这是可测、可换模型、可离线评测的根本。

### 2.2 数据流（单帧）
```
CameraService.startImageStream(YUV420 image)
  → FramePipeline.preprocess(image) → Uint8 RGB 192×192 (或 int8 量化输入)
  → PoseEstimator.infer(rgb) → List<KeyPoint>(17)
  → SignalExtractor.toSignals(keypoints) → FrameSignals{ shoulderY, elbowLateral, confs }
  → SignalFilter.smooth(signals) → FrameSignals(平滑后)
  → PushupCounter.update(signals) → CounterState{ count, phase: UP|DOWN, frozen }
  → UI rebuild(OverlayRenderer.draw(state, keypoints))
```

### 2.3 帧率解耦
推理是瓶颈（~20-40ms/帧），但 UI 需 60fps 流畅。**强制隔离**：
- 推理在独立 Isolate（或 Platform Channel 后台线程）
- 推理产出以最近一帧关键点供 UI 叠加；UI 按相机帧率重绘，关键点做线性插值或保持最近值
- 不允许推理阻塞 UI 线程

### 2.4 状态管理
推荐 Riverpod（或 Bloc），负责：相机权限/生命周期、推理引擎单例、会话状态（计数/计时/暂停）、设置（阈值可调）。

## 3. 分层模块规格

> 每个模块给出：**职责 / 接口契约 / 关键实现点 / 可测试点**。接口用 Dart 伪签名表达，字段命名即契约。

### 3.1 Platform Layer

#### `CameraService`
```dart
abstract class CameraService {
  Future<void> initialize({required CameraLensDirection facing});
  Stream<CameraImage> get imageStream;     // YUV420
  CameraController get controller;          // 供预览
  Size get previewSize;
  SensorOrientation get sensorOrientation;  // 旋转修正必需
}
```
- 关键点：**必须正确处理 Android sensor orientation**（多数 Android 横置 90°/270°）。Step 0 离线视频已定向，App 实时帧未定向——这是 App 端 Step 0（§3.3②）失败的头号原因。
- 可测试点：旋转修正后，竖屏持机时输出帧与 Step 0 视频帧方向一致（可用同一人对比）。

#### 推理 delegate
- 必须实现 **NNAPI delegate** 可开关，启动时探测可用性
- 提供 GPU delegate 作为备选
- 记录实际启用 delegate 到运行时配置

### 3.2 Pipeline Layer — `FramePipeline`
```dart
class FramePipeline {
  /// YUV420 + sensor rotation → 192×192 输入张量(按模型 dtype: float32 或 int8)
  TensorInput preprocess(CameraImage image, SensorOrientation orient, {int target=192});
}
```
- 流水线：旋转 → 中心裁剪/letterbox（与 Step 0 一致）→ YUV→RGB → resize → 归一化或量化
- **必须与 Step 0 的 `letterbox_frame` + `make_input` 逻辑保持一致**（同一 letterbox、同一量化），否则 App 端关键点质量会偏离 Step 0 基准
- 可测试点：喂同一帧，App pipeline 输出的 RGB 与 Step 0 Python 输出像素差 ≤ 容差（黄金帧测试，§7）

### 3.3 Inference Layer — `PoseEstimator`
```dart
class KeyPoint { final int index; final double x,y; final double confidence; ... }  // x,y 为原图像素坐标
class PoseEstimator {
  Future<void> load(ModelConfig cfg);       // assets/models/*.tflite
  List<KeyPoint> infer(TensorInput input);  // 17 点, 已映射回原图坐标(letterbox 逆变换)
}
```
- letterbox 逆变换必须复用 Step 0 的 `keypoints_to_pixels` 逻辑（pad/scale 还原）
- 可测试点：同一输入两次推理结果一致（确定性）；App 输出与 Step 0 离线输出坐标差 ≤ 容差

### 3.4 Domain Layer — 纯算法（可单测，零依赖）⭐

#### `SignalExtractor`
```dart
class FrameSignals {
  final double shoulderY;        // 主信号:加权肩Y, 左右肩按 conf 加权
  final double? headY;           // nose_y 备选(nose conf≥阈值时)
  final double? elbowLateral;    // 辅助:两肘横向间距 或 肘相对肩横偏
  final double shoulderConf, elbowConf, noseConf;
  final List<KeyPoint> raw;
}
```
- 加权均值逻辑**直接复用 Step 0 的 `weighted_mean`**（conf<min_conf 的点剔除，全 NaN 返回 NaN）
- 可测试点：给定构造的关键点，输出预期信号值（黄金数据）

#### `SignalFilter`
```dart
class SignalFilter {
  FrameSignals smooth(FrameSignals s);   // 滑动中值/均值, 窗口 N 帧
  void reset();
}
```
- 对 NaN（丢点）必须插值或保持，不得污染状态机
- 可测试点：注入抖动序列，输出平滑且无突变

#### `PushupCounter`（状态机）⭐核心⭐
```dart
enum Phase { up, down }
class CounterState { final int count; final Phase phase; final bool frozen; ... }
class PushupCounter {
  CounterState update(FrameSignals s);
  void reset();
  CounterConfig get config;  // 阈值/滞回/M/N 参数, 可运行时调整
}
```
- 算法见 §4.3，参数见 §4.4
- 可测试点：**这是整 App 准确率的根**，必须有完整单元测试套件（§7.1），用 Step 0 CSV 重放可得 5 计数

### 3.5 UI Layer
- `OverlayRenderer`（CustomPainter）：绘制骨架（连线阈值 conf≥0.2，与 Step 0 一致）、点（conf≥0.3 绿/<0.3 红）、左上角计数、状态标签、置信度告警
- 会话控件：开始/暂停/重置/结束；显示本次计数、计时
- 设置入口：阈值滑块（高级用户/调试用，第一版可隐藏在长按）

## 4. 核心算法规格

### 4.1 主信号定义（基于 Step 0 实测）
```
primary = shoulderY = weighted_mean([left_shoulder.y, right_shoulder.y],
                                    [left_shoulder.conf, right_shoulder.conf], min_conf=0.1)
辅助  = elbowLateral = abs(left_elbow.x - right_elbow.x)   # 标准俯卧撑肘外展, UP肘外/DOWN肘内
```
> Step 0 实测：本机位 Y 轴 UP态≈485 / DOWN态≈599，起伏 170px（画面高 13.3%）。**Y 越小=人撑得越高=UP**。

### 4.2 信号归一化
- 用**自适应相对量**而非绝对像素，适应不同人/距离/分辨率：
```
动态范围: 对近 N 帧主信号取 high = P80, low = P20 (滑动窗口, N≈90 帧≈3s)
归一位置: pos = (currentY - low) / (high - low)      # pos≈0→UP高位, pos≈1→DOWN低位
```
- 若 `high-low < height*0.04`（起伏不足），状态机冻结不计数

### 4.3 状态机（带滞回 hysteresis）
```
阈值:
  thr_down = low + 0.6*(high-low)   # pos>0.6 进入 DOWN 区
  thr_up   = low + 0.4*(high-low)   # pos<0.4 回到 UP 区   (0.4~0.6 为滞回死区,抗抖动)

切换规则:
  UP   → DOWN : primary ≥ thr_down, 连续 M_down 帧成立, 且 shoulderConf ≥ conf_thr
  DOWN → UP   : primary ≤ thr_up,   连续 M_up 帧成立,   计数 count += 1

丢点保护:
  shoulderConf < 0.3 的帧: 该帧不参与判定, 保持上一状态, 累计 frozen 计时
  连续 frozen > freeze_limit (≈1.5s): 状态机冻结,丢弃当前半周期,不计漏计误计

超时保护:
  进入 DOWN 后超过 cycle_timeout (≈5s) 未回 UP: 视为异常, 丢弃, 回到 UP 等待
```
> Step 0 实测用 0.4/0.6 分位滞回 + min_gap 8 帧得 5 周期，与帧事件自洽。App 端窗口与 M 值需用 §8 评测集重新标定，**禁止直接套用 Step 0 的特化值**。

### 4.4 可调参数表（`CounterConfig`）
| 参数 | 含义 | 初始值（待标定） | 标定依据 |
|---|---|---|---|
| `windowN` | 动态范围滑动窗口帧数 | 90 (≈3s@30fps) | 评测集 |
| `pHigh,pLow` | 动态范围分位 | 0.8, 0.2 | 评测集 |
| `thrDownPos,thrUpPos` | 滞回阈值(归一) | 0.6, 0.4 | 评测集 |
| `mDown,mUp` | 切换确认帧数 | 3, 3 | 快/慢子集 |
| `confThr` | 主信号置信门槛 | 0.3 | Step 0 分布 |
| `ampMinRatio` | 最小起伏(画面高比) | 0.04 | Step 0 |
| `freezeLimitMs` | 丢点冻结上限 | 1500 | 评测集 |
| `cycleTimeoutMs` | 单周期超时 | 5000 | 评测集 |

| 指标 | 目标 | 硬门槛 |
|---|---|---|
| 推理 FPS（中端机,骁龙6/7系） | ≥15 | ≥10，否则不验收 |

## 7. 测试策略

### 7.1 单元测试（Domain Layer，强制 100% 覆盖核心路径）
- `SignalExtractor`：构造关键点 → 预期信号（含丢点/全 NaN）
- `SignalFilter`：抖动注入、NaN 插值、窗口边界
- `PushupCounter`：⭐黄金测试 —— **重放 Step 0 的 `out_signals.csv`，断言最终计数 = 5**（与 Step 0 周期数一致）。这是状态机正确性的客观锚点。
- 参数化测试：快/慢节奏、丢点、起伏不足、超时，各构造序列断言行为

### 7.2 集成测试（Pipeline + Inference 一致性）
- 黄金帧测试：取 Step 0 视频的 N 帧，比对 App pipeline 输入张量与 Python 输入的像素差 ≤ 容差；比对关键点坐标差 ≤ 容差
- 目的：防止 App 端预处理（旋转/量化/letterbox）引入 Step 0 不存在的偏差

### 7.3 端到端
- 用预录视频喂入（离线回放模式，调试用），计数与人工标注对比
- 真机实时：至少 3 人实测，录屏 + 会话记录

## 8. 评测集与准确率验收

### 8.1 评测集要求（没有它，准确率无从谈起）
```
段数: ≥ 25 段真实视频
覆盖:
  - 人数 ≥ 5（含 ≥1 女 / 不同体型）
  - 速度: 慢(<1.5s/个) / 正常 / 快(<0.8s/个) 各占约 1/3
  - 疲劳: 每人含 1 段力竭段(动作变形但仍属标准俯卧撑)
  - 光照: 充足为主,含 2 段弱光
标注: 每段真实次数 + 每周期 UP/DOWN 起止帧(人工,双人交叉核对)
存储: evalset/<id>.mp4 + evalset/<id>.json
```
> 对抗式提醒：评测集**由验收方抽查至少 3 段重新数数复核**，不接受实现者自标。这是防"调参数调到自标集上"的根本。

### 8.2 离线评测协议
- 用 §7.3 的离线回放模式跑全集，输出 `signals.csv` + 计数
- 评测脚本 `eval.py` 输入标注 + 预测，输出：
```
MAE  = mean(|pred - truth|)
MAPE = mean(|pred-truth|/truth)
命中分布: ±0 / ±1 / 最差单段
分速度子集 MAPE(慢/正常/快)
```

### 8.3 准确率门槛（硬）
| 等级 | 标准 | 结论 |
|---|---|---|
| 合格 | MAPE ≤ 15% 且 最差单段 ≤ 2 且 快速子集 MAPE ≤ 25% | 第一版可交付 |
| 良好 | MAPE ≤ 8% 且 ±1 命中率 ≥ 80% | 产品可用线 |
| 不合格 | MAPE > 15% 或 快速子集崩 | 回 §4 排查 |
