## 0. 关键设计前提（用户拍板，已锁定）

- **帧率固定 30fps**：实时相机 sensorFps=30；离线视频重采样到 30fps。**算法基于 30fps 设计，不需兼容其他帧率。**
- **速度上限 2-3个/秒**（较快健身）：30fps 下每个俯卧撑约 10-15 帧。
- **已验证重采样不破坏计数**：视频3(50fps→30fps) 计数仍=5✅；视频5(60fps→30fps) 仍=1✅。

## 1. 任务目标

实现一个新的俯卧撑计数算法（峰谷检测），满足：
- **正确性**：视频1=5, 视频3=5, 视频4=3（已人工标注真值，重采样到30fps后验证）
- **抗噪声**：静止信号计 0；±20px 随机抖动计 0（不误计）
- **固定30fps**：所有输入统一30fps，算法不需处理帧率差异
- **支持较快速度**：至少数准视频4（30fps下72帧/3个，每个24帧/0.8秒，~1.25个/秒），尽量支持到2-3个/秒

## 2. 为什么换算法（根因）

当前 `pushup_domain.dart` 的 `PushupCounter` 是**滑动窗口百分位 + 滞回状态机**，缺陷是把时间量写成了固定帧数：

| 参数 | 当前值 | 本意（时间） | 问题 |
|---|---|---|---|
| `windowN` | 90帧 | 3秒 | 50fps下=1.8秒，72帧视频填不满 |
| `minGapFrames` | 8帧 | 0.27秒 | 不同fps下时间不同 |
| `freezeLimitMs→frames` | 45帧 | 1.5秒 | 帧率换算错配 |

**实测后果**：
- 视频3(50fps,343帧)：标定窗口吞掉开头第1个俯卧撑 → 得4（真值5）
- 视频4(30fps,72帧)：72帧<90帧窗口，永远不标定 → 得0（真值3）

**关键点没问题**：App 推理 vs Step0 的肩部Y中位差仅 4.4px（已用 `_app_v3.csv` 对比验证）。纯算法缺陷。

## 3. 已验证的关键结论（避免重复踩坑）

对"快速计数"和"抗噪声"做了参数搜索，确认不可调和：
- 要数准视频4（72帧/3个）→ 必须 mDown=mUp=1（单帧切换）
- 但 mDown=mUp=1 → ±20px噪声误计7个

### 3.2 峰谷检测思路对，但初版需调优
第一版峰谷检测（全局范围30%阈值 + 交替峰谷）：
- 视频1=5✅ 视频3=4❌ 视频4=2❌ 噪声=27❌
- 问题：全局范围阈值对噪声太敏感；峰谷交替逻辑有缺陷

### 4.3 重采样到30fps的方法（处理非30fps视频）
```python
def resample(rows, from_fps, to_fps=30):
    if from_fps==to_fps: return rows
    step = from_fps/to_fps; out=[]; i=0.0
    while i < len(rows): out.append(rows[int(i)]); i+=step
    return out
# 视频3: resample(load('step0/v3/out_signals.csv'), 50, 30) → 206帧
# 已验证: 重采样后 estimate_cycles 仍=5, 信号起伏336px
```
**App端实现**：ffmpeg抽帧时加 `-r 30` 即可重采样，见 VideoReplayService.prepare 的 args。

## 6. 算法设计建议（峰谷检测）

基于已验证的失败教训，建议方向：
1. **幅度阈值用局部范围，不用全局**：避免噪声的全局极值拉高阈值
2. **先平滑去噪**：肩部Y已有 window5 平滑，峰谷检测前可再降采样或中值滤波
3. **有效峰谷判据**：峰-谷高度差 > 阈值 且 谷-峰回升 > 阈值，且两者都满足才算1个
4. **阈值自适应**：基于信号自身统计（如标准差或局部range的比例），不硬编码像素值
5. **完全不依赖fps**：纯基于信号形状，帧数只用于索引不参与判定
