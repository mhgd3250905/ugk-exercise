# 俯卧撑完成检测 App — 实现计划与验收标准

> **文档定位**：这是给实现者的工程说明书，不是需求讨论稿。
> 设计原则：**第一性原理**（风险前置、最小成本探明最大不确定性）；
> 审查方式：**对抗式**（每一步都预设会失败，要求用数据证伪）。
>
> **角色分工**：同事负责实现 + 测试；验收方（本文档作者）按 §6 验收标准打分。
> 本文档随实现进展更新版本号。

---

## 0. 必须先读：本方案命门在哪里

整个项目 **90% 的风险集中在一个点**，与 Flutter、与状态机、与界面都无关：

> **MoveNet（或任何姿态模型）能否在「俯卧撑姿态 + 本次拍摄机位」下，稳定输出足够精度的关键点。**

姿态模型的训练集几乎全是「站立、正面/侧面、正常平视机位」。本次场景同时叠加：
1. **姿态异常**：人体水平横置（俯卧撑支撑姿态）
2. **机位异常**：地面、人前方，画面以头顶/肩部为主

这是双重 **分布外（OOD）**。模型在这一帧上的关键点置信度到底够不够，**不实测无法下结论**。因此本计划的第一步不是写 App，而是用半天跑 §2 的验证。验证不通过，整条技术路线要么换机位，要么换方案。

**结论先行**：本方案能否成立，取决于 §2 Step 0 的实测结果。Step 0 不通过，禁止进入 §3 工程实现。

## 1. 场景与机位校准（基于真实视频 俯卧撑.mp4）

### 1.1 视频元数据
```
分辨率：720 × 1280（竖屏）
帧率：30 fps
时长：12.0 s（共 359 帧）
编码：HEVC
```

### 1.2 机位确认（实现者第一步必须核对）
原始需求描述为「手机架在人前方地面，画面里只看到头顶 + 弯曲的手肘」。

**请实现者打开 俯卧撑.mp4 逐帧核对，确认以下三点并回填：**

| 待确认项 | 实测结论（实现者填） |
|---|---|
| 镜头相对人的方向（正前/正后/侧/顶上） | |
| 人物哪一面朝向镜头（脸朝镜头 / 后脑朝镜头 / 侧） | |
| 画面稳定可见的身体部位 | |

> ⚠️ **对抗式提醒**：机位决定了「哪些关键点可用」和「用什么信号计数」。这是地基。
> 务必以视频实际内容为准，不要以文字描述为准。若实际可见部位与预期不符，**立即停下并在 §1.3 记录偏差**，不要带着错误前提写代码。

### 1.3 计数信号的选择（取决于 §1.2 结论）

不同机位下，能用的信号不同。实现者据 §1.2 实测结论对号入座：

| 机位类型 | 主信号 | 辅助信号 | 说明 |
|---|---|---|---|
| 人前方、拍到头顶+手肘 | 头顶/肩的 **纵向 Y 位移** | 肘的**横向位移**（标准俯卧撑肘外展） | 纵向起伏不被纵深压缩，最稳 |
| 侧面平拍 | 肘角（2D 可见） | 肩 Y | 侧拍下肘角是干净信号 |
| 顶上俯拍 | 肩 Y（若可见） | — | 纵深被压，肘角基本失效，不推荐 |

> 默认按第一行（与本次视频一致的机位）实现。**主信号用「肩或头顶的 Y 坐标起伏」，肘作为辅助且仅在置信度达标时参与。** 不要单押肘角——DOWN 最低点肘最易被前臂遮挡丢点。

## 2. Step 0：可行性验证（第一性原理 / 最高优先级）

**目标**：用最小成本回答「这条路通不通」。半天完成。**不通过则止步，不进入 §3。**

### 2.1 交付物
一个离线 Python 脚本 `step0_verify.py`，输入 `俯卧撑.mp4`，输出：
1. `out_keypoints.mp4`：原视频 + 17 个关键点叠加 + 每点置信度
2. `out_signals.csv`：逐帧导出 `frame, nose_y, shoulder_y, elbow_y, elbow_x, wrist_y, [各点confidence]`
3. `out_signals_plot.png`：肩 Y、头顶/nose Y、肘角随帧数的曲线图

### 2.2 模型
- **MoveNet SinglePose Lightning**，`.tflite`，输入 192×192 RGB，输出 17 关键点
- 来源：`Google/TensorFlow` 官方或 `STMicroelectronics/movenet`（HuggingFace）
- 放 `step0/models/`，不随 App 打包（Step 0 仅离线验证用）

### 2.3 验收标准（Step 0 通过条件）⭐核心⭐
**逐项用数据判定，不允许「看起来还行」：**

| 编号 | 检查项 | 通过阈值 | 不通过的后果 |
|---|---|---|---|
| C1 | 肩关键点置信度 | 整段视频 **肩点平均 confidence ≥ 0.5** | 主信号失效，换机位或换方案 |
| C2 | 头顶/nose 可用性 | 至少其一平均 conf ≥ 0.4 | 主信号降级，需依赖肩 |
| C3 | 主信号起伏幅度 | 肩 Y 的 UP/DOWN **峰谷差 ≥ 画面高度的 5%**（即 ≥64px/1280） | 起伏太小，状态机无法分辨 |
| C4 | 肘关键点可用性 | 肘点平均 conf ≥ 0.3（作为辅助，要求放宽） | 肘不可用，纯靠主信号计数 |
| C5 | 周期可辨 | 曲线图能肉眼看出与动作一致的周期数 | 信号被噪声淹没 |

**判定规则**：C1、C3、C5 三项是硬门槛，**任一不满足 → Step 0 判为不通过**。C2、C4 是加分项，不达标可降级方案但可继续。

### 2.4 Step 0 不通过的应对决策树
```
C1(肩) 不达标 → 换机位重录（优先侧拍）→ 重跑 Step 0
C3(起伏) 不达标 → 检查是否机位太远/人太小 → 调距离重录
C5(周期不可辨) → 考虑换 BlazePose 或 MediaPipe Pose 做对照
全部试过仍不达标 → 放弃姿态方案，评估视频分类或 IMU 方案
```

> 验收方说明：**Step 0 报告需附原始 CSV 与曲线图，验收方会复核数据，不接受截图口头结论。**

## 3. 工程实现方案（仅当 Step 0 通过）

### 3.1 技术栈（锁定，不擅自换）
```
框架：Flutter（Android only，第一版）
相机：camera 插件
推理：tflite_flutter（MoveNet Lightning，assets/models/）
delegate：优先尝试 NNAPI/GPU delegate（见 §3.4）
状态机：PushupCounter（纯 Dart，可单测）
叠加层：CustomPainter 绘制骨架 + 计数
```

### 3.2 模块结构
```
lib/
  main.dart
  camera/camera_view.dart        # 相机预览 + 帧采集
  camera/frame_pipeline.dart     # YUV→RGB→resize→归一化
  pose/pose_estimator.dart       # 加载 tflite，输入帧→输出17点
  pose/keypoint.dart             # 关键点数据结构(含confidence)
  counter/pushup_counter.dart    # 状态机(纯逻辑,无UI依赖)
  counter/signal_filter.dart     # 滑动均值/中值 防抖
  overlay/overlay_view.dart      # 骨架+计数+状态绘制
```

### 3.3 实现顺序（严格按此序，每步可独立验证）
```
① 打通相机预览            → 验证：能看到画面
② 接通推理,打印关键点日志  → 验证：logcat 看到17点坐标与置信度
③ 绘制骨架叠加            → 验证：骨架贴合人体
④ 接 PushupCounter 状态机  → 验证：实时输出 UP/DOWN 状态切换
⑤ 加防抖 + 计数显示       → 验证：见 §6 验收
```
> 对抗式提醒：② 是 App 版的 Step 0。若 ② 阶段关键点置信度与离线 Step 0 差距大（通常因相机帧预处理/旋转/输入格式问题），**必须先解决，不得带病推进**。

### 3.4 性能要求
- 推理频率目标：**≥10 FPS**（实测，非理论）
- 预处理（YUV→RGB→resize）必须在 Dart 层高效完成，否则考虑 Platform Channel / FFI
- 必须测试 **NNAPI delegate** 开启与关闭的 FPS 差异，记录在验收报告
- 中端机型目标机（如骁龙 6/7 系）需达标；旗舰机仅作参考

## 4. 状态机与计数逻辑

### 4.1 输入信号
```
primary = shoulder_y 或 head_y（取 §2 实测更稳的那个），归一化到 [0,1]
elbow_lateral = 两侧肘点横向间距 或 肘相对肩的横向偏移（标准俯卧撑辅助）
各点 confidence
```

### 4.2 状态机
```
状态：UP | DOWN
阈值：
  ratio_high = primary 的近 N 帧(滑动)最大值
  ratio_low  = primary 的近 N 帧(滑动)最小值
  thr_down = ratio_low + 0.5*(ratio_high - ratio_low)   # 进入下压区
  thr_up   = ratio_low + 0.5*(ratio_high - ratio_low)   # 回到撑起区（第一版可同阈，加滞回更稳）

切换规则（带滞回 hysteresis）：
  UP   → DOWN : primary > thr_down   连续 M 帧 且 防抖后成立
  DOWN → UP   : primary < thr_up     连续 M 帧 且 防抖后成立
  DOWN → UP 完成时 count += 1
```
- 阈值用「自适应相对量」而非绝对像素，适应不同人/距离
- M、N 参数需用 §5 标注数据标定，**禁止拍脑袋定值**

### 4.3 丢点保护
- 主信号点 confidence < 0.3 的帧：该帧不参与状态切换判定，保持上一状态
- 连续丢点超过 1 秒：状态机冻结，不计漏计也不计误计

## 5. 评测数据集（验收的基础，没有它准确率无从谈起）

### 5.1 数据要求
```
段数：≥ 20 段真实视频
覆盖：不同人(≥3)、不同速度(快/慢)、不同疲劳程度
每段人工标注：正确次数 + 每 UP/DOWN 起止帧
格式：videos/xxx.mp4 + labels/xxx.json
```

### 5.2 评测指标（验收硬指标）
```
MAE  = mean(|预测次数 - 真实次数|)        # 平均绝对误差
MAPE = mean(|预测-真实|/真实)              # 平均相对误差
误差分布：±0 命中率、±1 命中率、最差案例
```

## 6. 验收标准（验收方据此打分）

### 6.1 交付物清单（缺一不验收）
- [ ] `step0_verify.py` + 模型 + 生成的 mp4/csv/png（Step 0 报告）
- [ ] Step 0 判定结论（通过/不通过 + 对应数据）
- [ ] Flutter App 可安装 APK
- [ ] 源码仓库（含 §3.2 结构）
- [ ] §5 评测集 + 评测脚本 + 评测结果表（MAE/MAPE/命中率）
- [ ] 录屏：App 实时计数演示（至少 3 段不同人）

### 6.2 功能验收（逐条 pass/fail）
| 编号 | 项 | 标准 |
|---|---|---|
| F1 | 相机预览 | 开启即显示，无卡顿 |
| F2 | 骨架叠加 | 关键点贴合人体，肉眼可见无明显错位 |
| F3 | 状态显示 | 实时显示 UP/DOWN，与动作一致 |
| F4 | 计数 | 完整 UP→DOWN→UP 计 1，单段演示正确率 ≥ 90% |
| F5 | 防抖 | 抖动/短暂丢点不产生误计 |
| F6 | 性能 | 中端机 ≥10 FPS，发热可接受 |

### 6.3 准确率验收（硬指标）
| 等级 | 标准 | 结论 |
|---|---|---|
| 合格 | MAPE ≤ 15% 且 最差单段误差 ≤ 2 次 | 第一版可交付 |
| 良好 | MAPE ≤ 8% 且 ±1 命中率 ≥ 80% | 达到产品可用线 |
| 不合格 | MAPE > 15% | 回到 §2/§4 排查，不验收 |

> 验收方将**独立复核** Step 0 数据与评测集至少 3 段抽样，不接受自报数据。

## 7. 风险登记表（实现中持续更新）

| 风险 | 概率 | 影响 | 触发条件 | 应对 |
|---|---|---|---|---|
| MoveNet 在本视角关键点不稳 | 高 | 致命 | Step 0 C1/C5 不达标 | 换机位/BlazePose/MediaPipe |
| DOWN 最低点肘丢点 | 中 | 中 | C4 不达标 | 主信号不依赖肘，§4.3 丢点保护 |
| 预处理慢导致 FPS 不达标 | 中 | 中 | ③ 阶段 FPS<10 | FFI/Platform Channel + NNAPI |
| 防抖参数导致漏计快动作 | 中 | 中 | 快速俯卧撑 MAE 高 | 用 §5 快速子集调 M/N |
| 用户动作不标准误判 | 中 | 中 | 非外展肘姿 | 第一版明确约束，文档告知用户 |

## 8. 给实现者的 3 条铁律

1. **Step 0 不过，不写 App。** 不要倒序。
2. **所有「看起来还行」都要有数据支撑。** 关键点稳不稳、FPS 多少、准确率多少，全部用数字说话，验收方按 §6 复核。
3. **机位和俯卧撑类型以视频实际为准，不以文字描述为准。** §1.2 必须先核对回填。
